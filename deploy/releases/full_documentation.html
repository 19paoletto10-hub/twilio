Przegląd rozwiązania

Twilio Chat App to dedykowane rozwiązanie dla firm, które potrzebują bezpiecznego, kontrolowanego kanału komunikacji SMS/MMS z możliwością automatyzacji i integracji z modelami AI. Produkt łączy prostotę obsługi z polityką audytowalności i pełnym zapisem danych.

Kluczowe funkcje

  • Dashboard & Wiadomości — szybki wgląd w wszystkie konwersacje, filtry, podgląd statusów i pełny audit trail.
  • Czat 1:1 — zakładka dedykowana pojedynczemu numerowi z inteligentnym przewijaniem, notatkami i wsparciem MMS.
  • Auto‑reply & AI Studio — przełączanie trybów odpowiedzi (szablon, AI, fallback bot) i testy bez wysyłki.
  • Przypomnienia — harmonogram cyklicznych SMS-ów z kontrolą statusu i logiem prób wysyłki.
  • Multi‑SMS — kampanie do wielu odbiorców: import numerów w dowolnym formacie, deduplikacja, raporty sukcesów/błędów.
  • News / FAISS / RAG — scraping artykułów, generowanie streszczeń, zarządzanie odbiorcami i backup indeksu.
  • API + webhooki — REST API do integracji z CRM i automatyzacją, w tym wysyłką, pobieraniem historii i zarządzaniem workerami.
  • Operacje & DevOps — docker-compose, healthcheck, logowanie strukturalne i rozpisany runbook prod.

Co widzisz

  • Przykład spójnego UI — jedna paleta kolorów, badge statusów, czytelne odstępy.
  • Elementy powtarzające się w innych zakładkach (karty, toasty, filtry).

Co możesz zrobić

  • Monitorować rozmowy na żywo, przełączać zakładki i diagnozować statusy.
  • Używać UI jako wzorca dla własnych integracji (np. zachować to samo nazewnictwo statusów).

Co dzieje się w tle

  • UI korzysta z tych samych endpointów REST, które udostępniamy integratorom.
  • Wszystkie interakcje są auditowane — layout ma na to wpływ (np. rozróżnienie inbound/outbound).

Szybki start (lokalnie)

Minimalny zestaw do uruchomienia systemu w trybie developerskim: Python, konto Twilio i podstawowa konfiguracja środowiska. AI/FAISS jest opcjonalne.

Wymagania
  • Python 3.10+
  • Konto Twilio (numer SMS lub Messaging Service)
  • (Opcjonalnie) OpenAI dla AI auto‑reply oraz RAG/FAISS
Uruchomienie 
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env

python run.py
Panel domyślnie działa pod adresem http://0.0.0.0:3000.
Webhooki Twilio

Po uruchomieniu ustaw w Twilio adresy webhooków na:

  • PUBLIC_BASE_URL/twilio/inbound (Incoming)
  • PUBLIC_BASE_URL/twilio/status (Status Callback)

Konfiguracja środowiska (.env)

Konfiguracja jest oparta o zmienne środowiskowe. System wymaga minimum danych Twilio; funkcje AI/FAISS aktywują się dopiero po ustawieniu klucza OpenAI.

Najważniejsze zmienne

# Twilio
TWILIO_ACCOUNT_SID=AC...
TWILIO_AUTH_TOKEN=...
TWILIO_DEFAULT_FROM=+48123456789
TWILIO_MESSAGING_SERVICE_SID=

# Aplikacja
APP_HOST=0.0.0.0
APP_PORT=3000
APP_DEBUG=true
DB_PATH=data/app.db
PUBLIC_BASE_URL=https://twoja-domena.pl
TWILIO_VALIDATE_SIGNATURE=true

# OpenAI – News/FAISS (RAG)
SECOND_OPENAI=sk-...
SECOND_MODEL=gpt-4o-mini
EMBEDDING_MODEL=text-embedding-3-large

Tryb dev vs prod

  • W prod ustaw APP_DEBUG=false i zostaw TWILIO_VALIDATE_SIGNATURE=true.
  • W dev możesz wyłączyć walidację podpisu tylko na czas testów lokalnych.

AI / FAISS

  • Brak SECOND_OPENAI wyłącza integrację OpenAI i ogranicza funkcje RAG/embeddings.
  • Modele można zmieniać bez modyfikacji kodu (zmienne środowiskowe).

Nadawca Twilio

  • Ustaw TWILIO_DEFAULT_FROM albo TWILIO_MESSAGING_SERVICE_SID.
  • Panel sygnalizuje brak konfiguracji nadawcy.

Architektura techniczna

Modularna aplikacja Flask z dedykowanymi modułami integracji i workerami. Architektura jest zoptymalizowana pod operacje (logi, przewidywalność, prosty backup) i minimalną liczbę zależności (SQLite jako główny store).

KomponentRola
`webhooks_bp`HTTP webhooki Twilio i API
`ai_service`Generowanie odpowiedzi AI i budowa kontekstu
`multi_sms.py`Worker batchowy dla kampanii Multi‑SMS i rezerwacja zadań
`twilio_client`Warstwa wysyłki wiadomości i zarządzania wątkami
`database.py`SQLite, migracje, helpery biznesowe
Najważniejsza idea

Panel WWW, webhooki Twilio, workery i operacje News/FAISS opierają się o wspólny magazyn (SQLite + katalog danych). Dzięki temu system jest audytowalny i odporny na restart.

Przepływy (end‑to‑end)

Poniższe scenariusze opisują „co dzieje się od kliknięcia do efektu” — w ujęciu operacyjnym i technicznym.

Inbound wiadomość (Twilio → system)

  1. Twilio wywołuje webhook POST /twilio/inbound.
  2. System waliduje podpis (produkcyjnie) i zapisuje wiadomość w SQLite.
  3. Jeśli aktywny jest auto‑reply (klasyczny lub AI), wiadomość trafia do kolejki workera.
  4. Worker generuje odpowiedź i wysyła przez Twilio; rezultat jest audytowany w historii.

Outbound z panelu (operator → klient)

  1. Operator wysyła wiadomość w UI (chat).
  2. Frontend wywołuje API wysyłki (SMS/MMS).
  3. System zapisuje rekord wiadomości i nadaje przez Twilio.
  4. Callback statusu POST /twilio/status aktualizuje delivery status w bazie.

News / RAG (scheduler → odbiorcy)

  1. Scraper pobiera treści do katalogu danych.
  2. Indeks FAISS jest budowany/aktualizowany (embeddings + mapowanie do dokumentów).
  3. O określonej godzinie scheduler generuje streszczenie (RAG) i wysyła do odbiorców.

Dashboard i lista wiadomości

Strona startowa przedstawia wszystkie aktywne wątki w jednym miejscu. Każdy wiersz zawiera: numer rozmówcy, skrót ostatniej wiadomości, kierunek (inbound/outbound), status dostarczenia i sygnaturę czasu. Filtry zachowują się tak samo w UI i API, dzięki czemu można je wykorzystywać w automatyzacjach.

Co widzisz

  • Wiersze z metadanymi (nadawca, skrót treści, status, data/godzina).
  • Kolumny dostosowujące się do szerokości ekranu — na mobile przechodzą w karty.
  • Ikony kanałów (SMS/WhatsApp) oraz badge informacyjne (np. failed).

Możliwe interakcje

  • Przejście do widoku czatu w trybie pojedynczej rozmowy (ikona strzałki).
  • Filtrowanie po statusie, dacie lub rodzaju kanału w celu szybkiej diagnostyki.
  • Pobranie danych do CSV przez API (te same parametry filtrów).

Co się zapisuje

  • Każda wiadomość trafia do SQLite z pełnym audit logiem statusów.
  • Numery normalizowane są do E.164, co scala rozmowy niezależnie od formatu.
  • Statusy aktualizowane są przez webhook POST /twilio/status.

Czat 1:1 — kontekst operacyjny

Po wejściu w szczegóły rozmowy UI przełącza się na layout bąbelkowy. Po prawej stronie dostępne są skróty (przełącz między trybami auto‑reply, podgląd historii statusów, szybkie szablony). Wysyłanie obsługuje SMS i MMS, przelicza segmenty i pokazuje z jakiego numeru/mes saging service skorzystasz.

  • Wiadomości ułożone chronologicznie, z datą, godziną i oznaczeniem osoby (operator/klient).
  • Załączniki MMS wyświetlane jako miniatury, które można otworzyć w nowym oknie.
  • Status wysyłki aktualizowany w czasie rzeczywistym dzięki pollingowi REST.

Auto‑odpowiedź (tryb szablonowy)

Globalny przełącznik auto‑reply pozwala błyskawicznie włączyć komunikat informacyjny. Zmiany są walidowane po stronie klienta i serwera, a worker natychmiast pobiera nową treść.

Co widzisz

  • Stan aktywacji oraz edytor wiadomości z licznikiem znaków.
  • Informacje o ostatniej automatycznej odpowiedzi (kiedy i do kogo).

Możliwe interakcje

  • Włączenie/wyłączenie trybu bez restartu aplikacji.
  • Zmiana treści (np. komunikat o awarii) z natychmiastowym zapisem.
  • Przejście do AI Studio, jeśli potrzebujesz inteligentniejszego trybu.
Aktywacja AI wyłącza tryb szablonowy, aby uniknąć dublowania odpowiedzi.

Co dzieje się w tle

  • Webhook inbound zapisuje wiadomość i dodaje zadanie do kolejki.
  • Worker auto‑reply wysyła odpowiedź tylko dla wiadomości nowszych niż moment aktywacji.
  • Każda odpowiedź jest audytowana w tabeli messages z flagą automatyczną.

Przypomnienia — kampanie cykliczne

Zakładka przypomnień zarządza kampaniami cyklicznymi (np. raporty dzienne, powiadomienia o płatnościach). UI prezentuje stan aktywacji, interwał, najbliższy termin oraz log wykonania.

Co widzisz

  • Tabelę z harmonogramami (numer, treść, interwał, aktywność).
  • Panel szczegółów — kiedy ostatnio i następnie zadanie będzie wysłane.

Możliwe interakcje

  • Dodanie nowego przypomnienia z walidacją formatu numeru i interwału.
  • Wstrzymanie/wznowienie oraz ręczne wymuszenie wysyłki.
  • Podgląd logów błędów, jeśli worker napotka problem (np. błąd Twilio).

Co dzieje się w tle

  • Worker cyklicznie sprawdza tabelę scheduled_messages.
  • Każda próba wysyłki jest zapisana w historii wiadomości i logach operacyjnych.
  • API /api/reminders obsługuje CRUD i akcje specjalne.

Multi‑SMS — wysyłka jednej wiadomości do wielu odbiorców

Zakładka Multi‑SMS łączy prostotę formularza z zaawansowaną kontrolą — operator może wkleić numery w dowolnym formacie, uruchomić kampanię i śledzić każdy odbiór w czasie rzeczywistym.

Jak to działa (w praktyce)

  1. Operator wkleja numery (przecinki / średniki / nowe linie) i treść wiadomości.
  2. Backend normalizuje numery i zapisuje batch + rekordy odbiorców w SQLite.
  3. Worker w tle rezerwuje najbliższy batch pending i przetwarza odbiorców.
  4. Każda próba wysyłki aktualizuje status odbiorcy oraz liczniki batcha (sukcesy/błędy).

Co widzisz

  • Formularz utworzenia batcha (treść + lista numerów).
  • Historię batchy z licznikami: wysłane / błędy / łącznie oraz status batcha.
  • Możliwość rozwinięcia listy odbiorców i podejrzenia statusów per numer.
  • Badge statusów (pending, processing, completed, failed).

Możliwe interakcje

  • Wklejenie numerów w różnych formatach (przecinki/linie) — system je rozbije, oczyści i znormalizuje.
  • Utworzenie batcha bez czekania na wysyłkę (wysyłka idzie w tle).
  • Kontrola postępu: odświeżanie historii batchy i rozwijanie odbiorców dla wglądu w statusy.
  • Diagnostyka błędów: podejrzenie error per numer (np. niepoprawny format, błąd Twilio, brak routingu).
  • Pobranie raportu przez API i eksport do CSV (np. dla działu marketingu).

Co dzieje się w tle

  • Backend tworzy batch i odbiorców w SQLite (kolejka DB oparta o statusy).
  • Worker rezerwuje batch do przetworzenia, wysyła SMS per odbiorca i aktualizuje statusy w tabelach multi_sms_*.
  • Każda próba wysyłki jest zapisywana w historii (messages), co daje pełny audyt.
  • UI pobiera batchy przez GET /api/multi-sms/batches (opcjonalnie wraz z listą odbiorców).
Optymalizacja UX
  • Formularz ma automatyczne przeliczanie liczby odbiorców oraz wstępne czyszczenie numerów (trim, usunięcie separacji, walidacja E.164).
  • Panel pokazuje skróty klawiaturowe i wskazówki (np. jak wkleić listę z Excela).
  • Wskaźnik postępu aktualizuje się asynchronicznie (polling co kilka sekund, limitowany by nie obciążać API).

Kontrola ryzyka

  • Limit odbiorców na batch: 250 (ochrona kosztów i przypadkowej wysyłki masowej).
  • Odporność na restart: stan utrzymywany w bazie, worker po restarcie kontynuuje kolejkę.
  • Audyt: każda wysyłka trafia do historii wiadomości oraz ma wynik per odbiorca.

News / FAISS (RAG) — scraping, indeksowanie i dystrybucja

Trzy zakładki News prowadzą operatora przez cały cykl życia informacji: od scrapingu i budowy indeksu po dystrybucję i backup. Każda sekcja ma jasne opisy, kontrolki i mechanizmy bezpieczeństwa (np. limit rozmiaru uploadu).

Co widzisz

  • Akcje administracyjne: test połączenia, wymuszenie scrapingu, podgląd plików źródłowych.
  • Komunikaty o ostatnim buildzie, liczbie artykułów i stanie indeksu.
  • Wskazówki jak często uruchamiać scraping i jakie kategorie są objęte.

Możliwe interakcje

  • Test generacji bez wysyłki (bezpieczne sprawdzenie jakości streszczeń i konfiguracji promptu).
  • Uruchomienie scrapingu „na żądanie” (ręczny refresh danych) oraz szybka weryfikacja, czy pliki zostały zapisane.
  • Pobranie plików wejściowych (JSON/TXT) do audytu jakości danych.

Co dzieje się w tle

  • Scraper pobiera treści i zapisuje pliki w katalogu danych (dla każdej kategorii osobno).
  • Po pobraniu danych budowany/aktualizowany jest indeks FAISS, aby zapytania RAG miały świeży kontekst.
  • API: POST /api/news/scrape, GET /api/news/files, GET /api/news/files/<filename>.

Co widzisz

  • Listę odbiorców powiadomień newsowych (numer + godzina + tryb).
  • Flagi enabled, informację o ostatniej wysyłce oraz tryb generacji.
  • Wskaźnik liczby artykułów wykorzystanych przy ostatnim wysłaniu (QA jakości).

Możliwe interakcje

  • Dodanie odbiorcy (numer w E.164) i wybór trybu: wszystkie kategorie (domyślnie) vs prompt niestandardowy.
  • Włączenie/wyłączenie odbiorcy oraz test wiadomości bez wysyłki (preview).
  • Force-send (wysyłka ręczna) dla odbiorcy w celach operacyjnych i weryfikacji routingu Twilio.
  • Eksport listy odbiorców do JSON przez API (np. backup lub audyt).

Co dzieje się w tle

  • Konfiguracja odbiorców jest zapisywana w pliku news_config.json.
  • Scheduler/worker wysyła o określonej godzinie wygenerowane streszczenia (jeśli odbiorca jest włączony).
  • Wysyłka jest realizowana przez Twilio i powinna być widoczna w historii wiadomości.
  • API: GET/POST/DELETE /api/news/recipients + akcje per odbiorca: toggle, test, send.

Co widzisz

  • Status indeksu (czy załadowany, ile wektorów, jakie modele embeddings/chat).
  • Gotowość backupu (wymagane pliki) oraz operacje import/export.
  • Informację o tym, który plik manifestu jest aktywny i kiedy ostatnio przeprowadzono backup.

Możliwe interakcje

  • Eksport backupu (ZIP) i import w celu odtworzenia środowiska (np. po wdrożeniu lub awarii).
  • Budowa indeksu z istniejących plików oraz test zapytań FAISS (diagnostyka jakości i „świeżości” danych).
  • Pobranie manifestu i zapisanie go do runbooka (kontrola zmian między środowiskami).

Co dzieje się w tle

  • FAISS przechowuje wektory embeddings + mapowanie do dokumentów.
  • Zapytania RAG pobierają kontekst z indeksu i opcjonalnie używają LLM do sformatowania odpowiedzi.
  • API: GET /api/news/faiss/status, GET /api/news/faiss/export, POST /api/news/faiss/import, POST /api/news/test-faiss.
Wskazówki operacyjne
  • Przed importem backupu sprawdź, czy środowisko ma wolne 250 MB (limit bezpieczeństwa).
  • Po imporcie UI wymusza odświeżenie indeksu w workerze, aby nowy zestaw dokumentów był aktywny.
  • Endpoint testowy POST /api/news/test-faiss pozwala ocenić jakość streszczeń przed wysyłką produkcyjną.

API modułu News (skrót)

  • Konfiguracja/odbiorcy: GET /api/news/config, GET/POST/DELETE /api/news/recipients, POST /api/news/recipients/<id>/(toggle|test|send)
  • Scraping: POST /api/news/scrape, pliki: GET/DELETE /api/news/files
  • FAISS: GET /api/news/faiss/status, GET /api/news/faiss/export, POST /api/news/faiss/import, POST /api/news/indices/build

Import backupu ma limit bezpieczeństwa 250 MB.

Dane (SQLite) — trwałość, backup i migracje

Aplikacja przechowuje stan w SQLite (domyślnie data/app.db). Schemat został zaprojektowany tak, aby wszystkie moduły (panel, API, workery) korzystały z jednego źródła prawdy.

Co jest zapisywane

  • messages: historia inbound/outbound, SID Twilio, status, błędy i timestamps.
  • multi_sms_*: batch + odbiorcy (statusy, liczniki, błędy per numer).
  • reminders: harmonogramy cyklicznych wysyłek.
  • news_config.json: konfiguracja odbiorców News (plik w katalogu danych).
  • settings: konfiguracja auto‑reply, AI i innych przełączników panelu.

Backup (praktycznie)

  • Najprostszy backup to kopia pliku DB (przy zatrzymanej aplikacji lub w oknie niskiej aktywności).
  • FAISS posiada dedykowany eksport/import ZIP z manifestem (API modułu News).
  • Przed aktualizacją schematu wykonaj migrację (skrypt manage.py db upgrade, jeśli włączone Alembic) lub snapshot pliku.
Retencja i compliance
  • Retencję wiadomości możesz kontrolować jobem czyszczącym (np. cron + skrypt SQL DELETE WHERE).
  • Indeks FAISS przechowuje embedowane teksty — dla RODO usuń dokumenty źródłowe i przebuduj indeks.
  • Każda tabela ma klucz created_at, co ułatwia audyt i raporty SLA.

API — integracja i automatyzacja

Panel WWW korzysta z REST API, które możesz również wywoływać z systemów zewnętrznych (CRM, marketing automation, chatboty). Poniżej skrót najczęściej używanych endpointów.

GET /api/conversations/<participant>Historia rozmowy (czat 1:1) — to samo, co widzi operator.
POST /api/send-messageWysyłka SMS/MMS; przyjmuje numer odbiorcy, treść, opcjonalnie media_url.

Autoryzacja i rate limiting
  • Webhooki Twilio używają podpisu X-Twilio-Signature; pozostałe endpointy mogą być zabezpieczone tokenem (Bearer) lub Basic Auth — zależnie od konfiguracji reverse proxy.
  • Wbudowane ograniczenia: 5 żądań/sek. na wskaźnik statusów (ochrona przed floodem), 1 kampania Multi‑SMS na minutę (domyślnie).
  • Dostępne formaty odpowiedzi: JSON (UTF-8). W przypadku błędów zwracany jest kod HTTP 4xx/5xx z opisem.

Glosariusz i najważniejsze pojęcia

Ten rozdział porządkuje terminologię spotykaną w panelach i API, aby PDF był samowystarczalny dla zespołów biznesowych i technicznych.

  • Batch (Multi‑SMS) — zadanie wysyłki jednej wiadomości do wielu odbiorców; posiada status (pending, processing, completed, failed) i liczniki skuteczności.
  • Recipient (Multi‑SMS) — pojedynczy numer przypisany do batcha; system śledzi status (pending / sent / failed) oraz ewentualny błąd Twilio.
  • Messaging Service SID — identyfikator grupujący numery w Twilio; jeśli skonfigurowany, Multi‑SMS i UI używają go jako nadawcy zamiast pojedynczego numeru.
  • Webhook — endpoint HTTP nasłuchujący żądań Twilio (np. inbound wiadomości, status dostarczenia). Każdy webhook jest podpisany nagłówkiem X-Twilio-Signature.
  • FAISS — biblioteka wektorowa używana do szybkiego wyszukiwania podobnych dokumentów, podstawowy komponent modułu News/RAG.
  • Scraper — proces pobierający artykuły i zapisujący je w katalogu danych; jego wynik zasila FAISS i moduł News.
  • Reminder — harmonogram cyklicznej wysyłki SMS. Każde przypomnienie opisuje numer docelowy, treść i interwał.
  • Auto‑reply — mechanizm automatycznych odpowiedzi (klasyczny lub AI). W trybie AI każde zapytanie jest przekazywane do modelu OpenAI z kontekstem rozmowy.
  • Normalized number — numer telefonu po oczyszczeniu (E.164). Pozwala scalać rekordy w historii i zapobiegać duplikatom.
  • News recipient — wpis w news_config.json; określa numer, godzinę wysyłki i tryb promptu (wszystkie kategorie vs. prompt niestandardowy).

AI & Auto‑reply — zasady działania

Zakładka AI Studio to centrum konfiguracji inteligentnych odpowiedzi. Pozwala wybrać model, zdefiniować prompt, ustawić parametry (temperatura, limit historii) i wykonać test bez obciążania budżetu SMS.

  1. AI buduje historię rozmowy z tabeli `messages` i wysyła zapytanie do OpenAI.
  2. Odpowiedzi są wysyłane tak, aby zachować wątek konwersacji (preferowana metoda `send_reply_to_inbound`).
  3. Operator może przetestować model w zakładce AI bez wysyłania SMS (endpoint `POST /api/ai/test`).

Co widzisz

  • Konfigurację AI (włączenie, model, prompt systemowy, temperatura, limit historii).
  • Podgląd kontekstu w formie bąbelków (ostatnie wiadomości, które trafią do LLM).
  • Panel testowy, który pozwala zweryfikować odpowiedź i ewentualnie ją poprawić przed wysyłką.

Możliwe interakcje

  • Włączenie AI auto‑reply i dostosowanie promptu pod głos marki (tone of voice, disclaimers).
  • Test generacji bez wysyłki (tekst lub JSON) oraz porównanie z odpowiedzią klasyczną.
  • Eksport konfiguracji promptu do JSON (przez API), aby dzielić się ustawieniami między środowiskami.

Co dzieje się w tle

  • AI buduje kontekst z historii wiadomości i wywołuje OpenAI wg ustawień (model/temperatura/limit historii).
  • Jeśli tryb auto‑reply jest aktywny, inbound webhook enqueue’uje zadanie odpowiedzi (bez blokowania requestu).
  • Wysyłka odpowiedzi przez Twilio jest rejestrowana w messages; statusy mogą się aktualizować callbackiem.
  • System unika odpowiedzi na starsze wiadomości sprzed aktywacji trybu (brak retro‑odpowiedzi).
  • UI regularnie sprawdza status klucza OpenAI przez GET /api/openai/status i komunikuje ewentualne braki.

Bezpieczeństwo — kluczowe zasady

System jest zbudowany tak, aby minimalizować ryzyko nieautoryzowanych wywołań webhooków, wycieków kluczy oraz niekontrolowanej wysyłki.

Webhooki Twilio: walidacja podpisu

  • Callbacki Twilio są weryfikowane nagłówkiem X-Twilio-Signature (RequestValidator).
  • Walidację można wyłączyć tylko dla testów lokalnych przez TWILIO_VALIDATE_SIGNATURE=false (produkcyjnie: zostawić włączoną).

Klucze i konfiguracja

  • Dane dostępowe Twilio są pobierane ze zmiennych środowiskowych i nie powinny trafiać do repozytorium.
  • DB jest plikiem (SQLite) — w produkcji zadbaj o uprawnienia do katalogu data/ oraz backup.

Kontrola kosztów

  • Multi‑SMS ma limit odbiorców per batch.
  • News wysyła wiadomości tylko do włączonych odbiorców; testy mogą działać bez wysyłki.

Logi & monitoring

Operacyjne utrzymanie aplikacji opiera się na logach strukturalnych, healthchecku HTTP i prostych metrykach liczonych w SQLite. Poniżej zalecenia dla środowisk produkcyjnych.

Logowanie
  • Flask loguje w formacie JSON (timestamp, poziom, moduł, komunikat).
  • Workery (auto‑reply, reminders, multi‑sms, news) raportują start/stop oraz błędy w osobnych namespace'ach.
  • Polecenie docker compose logs -f web lub integracja z Elastic/Splunk.
Monitoring
  • Healthcheck: GET /api/health (wykorzystywany przez Docker i load balancer).
  • Walidacja klucza OpenAI: GET /api/openai/status.
  • Proste metryki (liczba wiadomości, błędów) można odczytać zapytaniami SQL lub endpointami batchowymi.
Alerting
  • Konfiguruj alert, gdy healthcheck zwróci != 200 przez >3 próby (grace period 90s).
  • Monitoruj liczbę rekordów ze statusem failed w tabeli messages; próg zależny od biznesu.
  • Logi z workerów zawierają task_id — powiąż je z narzędziami observability dla szybkiej diagnozy.

Operacje — uruchomienie i utrzymanie

Aplikacja uruchamia workery w tle (m.in. Multi‑SMS, przypomnienia, News). Najważniejsze operacyjnie jest przewidywalne środowisko, logi i backup danych.

Środowisko

  • DB: DB_PATH (domyślnie data/app.db)
  • Twilio: TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN, oraz TWILIO_DEFAULT_FROM lub TWILIO_MESSAGING_SERVICE_SID
  • AI/FAISS: konfiguracja OpenAI (jeśli używana) + pliki indeksu w katalogu danych

Diagnostyka

  • Jeśli Multi‑SMS nie wysyła: sprawdź logi workera oraz czy tabele multi_sms_batches i multi_sms_recipients istnieją w DB.
  • Jeśli News/FAISS nie działa: sprawdź status GET /api/news/faiss/status i czy scraping zbudował indeks.

Backup

  • SQLite: kopia pliku DB + rotacja (np. dzienna).
  • FAISS: eksport ZIP przez endpoint i/lub zrzut katalogu indeksu.

Deployment & generowanie PDF

Plik HTML jest przygotowany do konwersji do PDF. Dla spójnych rezultatów rekomendowane jest użycie silnika drukowania z obsługą CSS @media print.
